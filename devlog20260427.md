### 3. SearXNG self-host **Railway-en** (frissített álláspont — holnapi munka!)

**Eredeti hibás álláspontom**: 04-26 este azt írtam, hogy SearXNG-t Railway-re felrakni nincs értelme, mert "ugyanaz az IP-pool". Ez **csak akkor lett volna igaz, ha a SearXNG-t pusztán DDG-re konfiguráljuk**. A friss diagnosztika fényében (burst-throttle, NEM IP-blokk) ez a kép más.

**Korrigált álláspont**: SearXNG **mehet Railway-re**, és **érdemes is**. Konkrét értéke:

1. **Multi-engine forgalom-szétosztás**: a SearXNG egyszerre hívja DDG, Brave, Bing, Wikipedia, Reddit, Mojeek, Wolfram engine-eket. Egy 60-query-s task megoszlik 7-8 engine közt → **~8 query/engine**, bőven a DDG-burst-küszöb alatt akkor is, ha Railway-IP-ről jön.
2. **API-kulcsos engine-ek IP-függetlenek**: Brave + Bing key-ből autentikál, az IP irreleváns. A scrape-engine (DDG, Mojeek) örökli a Railway-IP-problémát, de már csak a forgalom 1/8-án.
3. **Két szintű cache**: Bridge cache (10min/100 entry) + SearXNG saját cache.
4. **Aggregált találatok**: dedup + reranking, egyetlen engine-hiány nem lefagyaszt.
5. **Sima HTTP service**, nincs MCP-protokoll-réteg (a `brave-mcp-server` korábban ezért volt nehezen integrálható). `GET /search?q=...&format=json` — egy REST helper a Bridge-en.

**Ami NEM old meg**: a strukturált statisztikai adatok (HDI, PISA, Eurostat-számok) — azokra a 2. TODO (strukturált adat-API tool-réteg) az igazi válasz. SearXNG hír/narratíva-feladatokra való.

**Erőforrás**: SearXNG Docker image ~150-200MB, RAM 256-512MB, alacsony CPU. Railway-en kis service, kb. $2-3/hó tartomány.

**Hozzáállás**: standard `searxng/searxng` image + saját `settings.yml` Railway env varban vagy mounted volume-ben. **Saját fork NEM kell** — code-szintű custom plugin/ranking-szabály jelenleg nincs igény.
